## **Project Overview**

## **Project name:** DINO (大脳) A play on the individual kanji  with a meaning resembling the slang big brain.

## **One-sentence idea:** Input vocabulary in target language and store that in a list with information such as meaning in native language, confidence in understanding, date of addition, and create flash card tests.

## **Problem this solves:** In your daily life it may be inconvenient to keep a list of new words you encounter in a journal or the notes app. It would be convenient if it saved and kept track of these vocab and gave you the translation at the same time. Revisiting the words for review will also be convenient and easy to do.

## **Who is it for (target users)**: Language Learners, first iteration will be Japanese and English.

## **Why now (urgency / timing)**: Despite a wide array of options of study apps, this specific niche doesn’t seem to exist yet.

### **1\. `users` (The Core Accounts)**

*Stores app users. For the initial proof of concept (POC), this will contain a single hardcoded guest profile row.*

* `user_id` (Primary Key, string/UUID)  
* `email` (string)  
* `date_created` (timestamp)

### **2\. `lists` (The Custom Folders / Playlists)**

*Collections of vocabulary created by users. A list belongs to a single user.*

* `list_id` (Primary Key, string/UUID)  
* `user_id` (Foreign Key pointing to `users.user_id`)  
* `list_name` (string) — *e.g., "Niseko Ski Vocab"*

### **3\. `words` (The Global Dictionary Cache)**

*The shared master dictionary. Stores translation pairs globally to prevent duplicate translation API bills.*

* `word_id` (Primary Key, string/UUID)  
* `user_id` (string) — id of user creating row, if successful translation “system”  
* `input_word` (string) — *The text exactly as typed by the user*  
* `translated_word` (string) — *The output generated by the API, or manually typed if API fails*  
* `source_lang` (string) — *The detected source language code (e.g., `JA`, `EN`)*  
* `target_lang` (string) — *The requested target language code (e.g., `EN`, `JA`)*  
* `is_verified` (Boolean) — *`true` if successfully translated via DeepL/Google; `false` if the user typed it themselves because the API couldn't find a match*

### **4\. `list_words` (The Folder Junction Layer)**

*A completely "dumb" connector table that assigns a word to a list. Contains zero statistics.*

* `list_word_id` (Primary Key, string/UUID)  
* `list_id` (Foreign Key pointing to `lists.list_id`)  
* `word_id` (Foreign Key pointing to `words.word_id`)

### 

### **5\. `user_word_mastery` (The Universal Brain Layer)**

*Tracks the user's personal learning analytics globally. A user has exactly ONE relationship/fluency score with a word across the entire application.*

* `mastery_id` (Primary Key, string/UUID)  
* `user_id` (Foreign Key pointing to `users.user_id`)  
* `word_id` (Foreign Key pointing to `words.word_id`)  
* `confidence_rating` (integer, 1–5 scale) — *Used directly by the Flashcard engine*  
* `last_reviewed_date` (timestamp) — *Used for Spaced Repetition calculation*

## **Core Application Logic**

### **The "Find or Create" Look-up Rule:**

When a user searches for a word to add to a list, your backend/local code must follow these strict queries:

1. **Check Cache:** Query the `words` table where `input_word = user_text` AND `source_lang = detected_lang` AND `is_verified = true`.  
2. **Cache Hit:** Grab the existing `word_id`. **Do not call the translation API.**  
3. **Cache Miss:** Call the Translation API. Save the new text to `words` with `is_verified = true`.  
4. **API Failure Fallback:** If the API returns nothing (slang/typo), allow the user to type their own meaning. Save it to `words` with `is_verified = false`.  
5. **Link:** Insert a row into `list_words` to place the word in their folder. Ensure a row exists in `user_word_mastery` with a default confidence rating of `1`.

## **Risks, Concerns, & Development Watch-outs**

### **1\. The Japanese IME "Enter" Bug (High Risk for UI/UX)**

* **The Risk:** In Japanese keyboard software (IME), users hit the "Enter" key once to confirm their Kanji character selection, and a second time to submit the text field. It is incredibly common for apps to accidentally trigger the translation submit event on that very first "Enter" tap.  
* **How to Handle:** Submit input through a button press not an “Enter” key

### **2\. The `is_verified` Query Leaks**

* **The Risk:** If your database select query accidentally forgets to filter by `is_verified = true` during global look-ups, a user typing a broken word or an incorrect custom translation (`is_verified = false`) will accidentally leak into the app and become the standard translation definition for all future users.  
* **How to Handle:** Write a strict database helper function or security policy (like Supabase RLS) that makes it impossible to select `is_verified = false` unless the executing `user_id` matches the person who created it.

### **3\. Cascading Deletes on Playlists**

* **The Risk:** If a user deletes a list folder, you want to remove the playlist, but you **absolutely do not** want to drop the user's master vocabulary data or wipe out the global dictionary.  
* **How to Handle:** Ensure foreign key constraints on `list_words` use `ON DELETE CASCADE` for `list_id`. However, the `words` and `user_word_mastery` tables should remain completely untouched. If a word is deleted from a list, it should still live safely in their "universal brain" tracking database.

**4\. Look up Efficiency**

To avoid looking up the entire table for reading one list for example, create an index system.

CREATE INDEX idx\_list\_words\_list\_word ON list\_words(list\_id, word\_id); CREATE INDEX idx\_mastery\_user\_word ON user\_word\_mastery(user\_id, word\_id);
