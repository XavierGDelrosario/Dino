# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project

DINO (大脳) is a vocabulary-learning app (POC, Japanese ↔ English): translate words, save them to lists, later review via flashcards/spaced repetition. `DinoPOC.md` is the product brief and data-model rationale.

> Stack drift: `DinoPOC.md` predates code and says React Native + Firebase + DeepL. The actual stack is **Vite + React 18 + TypeScript + Supabase (Postgres/Auth/RLS) + a `translate` Edge Function**. The primary dictionary provider is **JMdict** (self-hosted, wired into the edge function); the machine-translation fallback for words JMdict lacks + the whole-paragraph gloss is **Google Cloud Translation v2**, enabled by the `TRANSLATION_API_KEY` edge secret (code done; needs a real key for live verify). **Code is the source of truth.**

The service layer is built and **compiles/builds green** (`tsc` + `vite build`). UI is underway — `App.tsx` is a session gate + tab nav (Translate · Lists · Review) over three surfaces wired to the services, plain React hooks + CSS (no router/state lib):
- **Translate** (`views/TranslateView.tsx` + `hooks/useTranslate.ts`): ONE input — **kuromoji decides word vs sentence** (`isSingleWord` in `analyze.ts`), no manual toggle. Single word → all senses (save defaults to the **primary**, others behind an "Other meanings" disclosure, to avoid cluttering ALL); inflected input resolves via its lemma (行った→行く). Sentence → the word-by-word reader (`components/translate/ParagraphReader.tsx`): grammatical tokens (particles 助詞 / auxiliaries 助動詞, via POS) render as plain text, not vocabulary; content words are colored by knowledge (grey=no entry · blue=addable · **red→green by confidence** once saved). **Hover** lists EVERY sense, each with its own `＋` — so a homograph (辛い → からい/つらい) lets you add the exact meaning, not just the primary; saved senses show `✓ n/5` (click = "don't know" lapse). An **Add all** button saves the primary of every new word. Per-sense saved/confidence state is keyed by dictionary `wordId` in `useTranslate` (shared by word + sentence modes). The whole-sentence gloss is served by the MT fallback (Google) when `TRANSLATION_API_KEY` is set.
- **Lists** (`views/ListView.tsx` + `components/lists/ListRow.tsx` + `hooks/useLists.ts`): ALL + sub-list chips, create/delete sub-list, inline edit-meaning (override), delete-from-vocab, tag/untag a word, add custom word.
- **Review** (`views/FlashcardView.tsx` + `hooks/useReview.ts`): the spaced-repetition flashcard loop.

Product/UI intent (hand-drawn) lives in `POC_Features.pdf`. UI surfaces now cover the POC sketch; the MT fallback (whole-sentence gloss, Google) is wired — remaining gaps are polish (per-kanji furigana, etc.).

**kuromoji runs in the browser — and getting it there needed two Vite fixes (`vite.config.ts`), both load-bearing:** (1) alias Node's `path` → `src/shims/path.ts` (kuromoji's loader calls `path.join`, which Vite externalizes → silent fallback to `Intl.Segmenter`, no lemmas); (2) a `serveDictRaw` plugin serves `/dict/*.dat.gz` as raw octet-stream with **no `Content-Encoding`** (Vite otherwise sets `Content-Encoding: gzip`, the browser auto-decompresses, and kuromoji's own gunzip dies with "invalid file signature"). Without both, JA segmentation silently degrades and inflected/compound words mis-resolve (e.g. 行った → bare 行). Verified in a headless browser.

## Architecture — the big picture

**Translation is backend-only.** The frontend `translate()` (`services/translation/client.ts`) only `invoke`s the `translate` edge function, which runs with the service role in `supabase/functions/translate/index.ts`. The browser can never translate on its own, and **only that backend can write `is_verified = true`** (RLS forbids clients). The **primary** provider is JMdict, queried server-side via the `jmdict_lookup` SQL function; `callTranslationProvider` is the MT fallback (**Google Cloud Translation v2**) for words JMdict lacks + the whole-paragraph gloss. Adding/swapping the MT provider is a body change to that one function (provider-agnostic `TRANSLATION_API_KEY` / `TRANSLATION_API_URL` secrets); nothing client-side changes.

**`persist` flag** (on the translate request): individual **words are cached** as verified (`persist: true`, default); a **whole-paragraph contextual translation is display-only** (`persist: false`, never stored — we won't keep thousands of unique paragraphs).

**Translation is backend-only, but morphological analysis is client-side.** Segmentation + readings/lemmas come from `services/language/analyze.ts` (`analyze()`), which routes Japanese to a lazily-loaded **kuromoji** engine (other languages → plain `Intl.Segmenter`, no reading/lemma). This runs in the browser because it feeds the ephemeral, display-only paragraph path — nothing analyzed is persisted. `analyze()` is the single swap point if it ever moves server-side. kuromoji readings are **best-effort** (statistical IPADIC; can misread ambiguous tokens) — distinct from the **authoritative** verified readings stored on `words`.

**Dictionary stack — JMdict vs `words` vs kuromoji (who answers what).** Three pieces, two reading "surfaces":
- **JMdict (`jmdict_*` tables)** — the authoritative SOURCE (full dataset, server-only). Holds writings, readings, senses, glosses. Queried ONLY by the edge function via `jmdict_lookup(input, source, target)`; clients can't touch it.
- **`words`** — the lazy verified CACHE the client reads. The edge function fills it from JMdict on a miss; each row carries its sense's reading inline (`input_reading`/`translation_reading`).
- **kuromoji** — a client-side morphological ENGINE (not data, not authoritative) for free text: segments space-less Japanese sentences and gives a best-effort reading + lemma per token.

Reading/furigana resolution depends on whether there's CONTEXT:
- **No context** (single-word lookup, flashcard): use the **`words` reading** — authoritative. kuromoji is unreliable on isolated fragments and is NOT used here.
- **With context** (sentence/paragraph, via `translateParagraph`): use **kuromoji** (context disambiguates homographs). It overrides a token's kuromoji reading with the `words` reading ONLY when **(a)** the surface IS the dictionary form (`token.text === lemma`, so a conjugated 行った keeps its surface reading いった, not the lemma's いく) **and (b)** the looked-up senses agree on a SINGLE reading (so a homograph like 辛い → からい/つらい defers to kuromoji's context). No extra query — it reuses the per-token `words` lookup done for meanings.

Translate flow (edge function), one word: **(1)** read all verified `words` senses → cache hit returns them (readings included), no JMdict query; **(2)** miss → `jmdict_lookup` returns meaning + reading together in one query, projected into `words`, returned as `words: Word[]` (`word` = primary); **(3)** MT fallback (Google) only if JMdict has no match (cached as a `mt:<input>` row, no readings). Homographs that JMdict splits into separate entries (辛い→からい/つらい) become separate `words` rows with their own correct readings — never swapped.

**`input` is the canonical headword, not the search term** — the kanji writing, EXCEPT for `uk` ("usually kana") entries which headline as kana with the kanji in `input_reading` (see the uk item below). `jmdict_lookup` (JA→EN) returns a `writing` per entry; the edge function stores THAT as `words.input` with the annotation in `input_reading`. So a hiragana search (ねこ) keeps the kanji (猫), and homophones split into their own kanji (はし → 橋 / 箸 / 端) instead of collapsing. Because the stored `input` may differ from what was typed, the edge function's cache read (`fetchVerified`) matches **`input` OR `input_reading`** so a kana search still resolves the kanji rows. `input_reading` is now NULL for kana-only words (no furigana needed). The UI shows `word.input` as the headword (`result__head`) so the kanji is visible. (Client-side `repository.ts` reads still match `input` exactly — a kana search misses the local cache and round-trips to the edge, which resolves it; an acceptable POC inefficiency.)

**Translate ≠ save — two distinct flows:**
- `services/lookup.ts` (**READ**): `lookupWord` (returns *all* meanings of a word — the first may be wrong, user picks) and `translateParagraph` (whole-paragraph display translation + reading/lemma-enriched tokens via `analyze()` + a `word → meanings` Map; meanings are looked up by **lemma** so conjugations like 行った resolve via 行く, then re-keyed under the surface text). Saves nothing to the user's lists.
- `services/dictionary.ts` (**WRITE / "quick add"**): `addWordToList` / `addWordsToList` translate and immediately save, accepting the preferred meaning.
- Saving a chosen word is **explicit**: `saveDictionaryWord` (`services/words/userWords.ts`) — it creates the user's `user_words` entry. Creating/editing a user's own word and deleting a word also live there.
- Rule of thumb: a function in `lookup.ts` never writes to a user's vocabulary; a verb-of-intent name (`lookup`/`translate`) whose body persists is a smell — split it.

**Auth = Supabase anonymous guest.** `session.ts ensureSession()` signs in anonymously (real `auth.uid()`, no login screen — the POC "guest profile") and ensures a `public.users` row. Every service is keyed on `userId`; RLS is keyed on `auth.uid()`.

## Data model

Tables in `supabase/migrations/20260613_init.sql` (one consolidated migration). The model **separates the global dictionary from each user's personal vocabulary**:
- `users` — `user_id` = Supabase Auth UID (TEXT). RLS: own row only.
- `lists` — a user's **sub-lists** (folders). `UNIQUE (user_id, list_name)`. "ALL" is **not** a stored list (see Invariants).
- `words` — **global dictionary cache**, verified + system-owned, **read-only to clients** (only the edge function writes). `UNIQUE (input, translation, source_lang, target_lang, is_verified)`; one input may have several sense rows. `input_reading` / `translation_reading` (nullable) hold the pronunciation reading of whichever side is non-phonetic — kana furigana for JA, pinyin for ZH; one side populated for JA↔EN, both for JA↔ZH, NULL otherwise. They are deterministic attributes of the headword, so **not** part of the `UNIQUE` key. The edge function fills them from JMdict's kana when projecting a sense; they are the furigana source for the no-context surface (lookups/flashcards) and are NULL only for words JMdict lacks (or before the ingest is run). (No `created_by`: every row is system-created by construction.) Also carries `part_of_speech TEXT[]` (JMdict POS), `frequency INT` (corpus rank — the DIFFICULTY axis, resolved by `services/difficulty`) and `difficulty_override INT` (curated 1..5, JLPT/HSK; NULL today) — all nullable, intrinsic, NOT part of the `UNIQUE` key. **`words` is considered schema-complete for the DinoMVP roadmap**: a column belongs here only if it's GLOBAL + INTRINSIC-to-the-sense + SCALAR; per-user data lives in `user_words`, and the relatedness/embeddings axis ships as a separate `word_embeddings` table (deliberately NOT a column here, so re-embedding never churns `words`).
- `user_words` — **a user's personal vocabulary**: one row per word they have. It references a dictionary sense (`dictionary_word_id`), **overrides** it (`custom_translation`), or **stands alone** (a created word: `custom_translation` set, no `dictionary_word_id`). Resolved meaning = `custom_translation ?? words.translation`. Readings are **read-only dictionary attributes resolved by joining `words`** (never stored on `user_words`, no custom readings): the input reading always comes from the sense; the **translation reading is suppressed on override** (the shown term is then the user's own, which the dictionary reading doesn't annotate); standalone created words have none. Mastery/review state lives here: `stability` (memory strength in days, NULL until first reviewed — the forgetting-curve parameter), `confidence_rating 0–5` (a **derived** display bucket of `stability`, written by `record_review()`), `last_reviewed_date`, `originally_translated_date`. The review schedule is a **continuous forgetting curve, not a stored due-date** — the queue ranks by live retrievability (see `services/review.ts`).
- `review_log` — **append-only** review history (`user_word_id`, `grade`, `reviewed_at`, `elapsed_days`, `prev_stability`, `new_stability`). Written ONLY by `record_review()` (now **`SECURITY DEFINER`**, with an explicit `user_id = auth.uid()` ownership guard since it bypasses RLS); RLS is **read-own SELECT only, NO client INSERT grant**, so the FSRS training history can't be forged. Kept because FSRS trains on it (impossible to backfill).
- `user_limits` (migration `20260620_user_limits.sql`) — **per-user RESTRICTIONS / entitlements** keyed on `user_id`. Columns: `paragraph_char_limit` (max chars per paragraph translation — the per-request guard) and `monthly_char_quota` (hard ceiling on chars translated per calendar month — the free-tier gate); NULL = use the app default. **Extensible**: each new restriction (voice, camera) is a new nullable column. RLS: **read-own, NO client write** (a client can't raise its own cap) — only the service role administers limits. A user with **no row** gets the code defaults, so the guest needs zero setup; keyed on `user_id`, so anonymous guests and future authed users (same uid on upgrade) share it.
- `translation_usage` (same migration) — **cumulative paid-MT characters per user per calendar month** (`user_id`, `period_month`, `chars_used`), the running total the monthly quota checks against. Written ONLY by the edge function (service role) via **`consume_translation_quota(user_id, chars, quota)`** — an atomic, per-user-advisory-locked **reserve-before-call** RPC that checks the quota AND increments in one step (returns `allowed`/`used`), so concurrent requests can't both pass a stale read and overshoot (the old check-then-meter race is gone). The edge calls it BEFORE the Google call, so a denied request costs nothing. Month bucket is computed in **UTC** on both sides (`date_trunc('month', now() AT TIME ZONE 'UTC')` in SQL; UTC in the edge/client) so they always agree. RLS **read-own, no client write** (can't fake/reset usage). A new month is a new row — no reset job.
- `list_words` — junction tagging `user_words` into sub-lists (`list_id ↔ user_word_id`). RLS: creating a tag requires **both** the list and the word to be the caller's (`WITH CHECK` on both FKs); removal only needs the list.

Cascade: deleting a `user_words` row removes the word from **every sub-list** (`list_words` cascades) — it leaves the user's whole vocabulary, and re-adding later starts fresh at `confidence_rating = 0`. Deleting a sub-`list` drops only its `list_words` tags; the `user_words` rows survive. The global `words` row is never touched by client deletes.

**JMdict source** (`supabase/migrations/20260618_jmdict.sql`): the authoritative dictionary is self-hosted in normalized `jmdict_entries` / `jmdict_kanji` / `jmdict_kana` / `jmdict_senses` / `jmdict_glosses` tables, loaded once by `scripts/ingest-jmdict.ts`. These are **server-only** (RLS on, no policies, no grants — only the edge function's service role reads them, via the `jmdict_lookup(input, source, target)` SQL function). `words` stays the **lazy verified cache**: on a miss the edge function calls `jmdict_lookup`, projects the matched senses into `words`, and returns them. **Readings ride inline on each `words` row** (`input_reading` / `translation_reading`) — that is the furigana source for the **no-context** surface (single-word lookups, flashcards), where kuromoji is unreliable in isolation. **Sentence/paragraph furigana uses client-side kuromoji** (context-aware); `translateParagraph` only overrides a token's kuromoji reading with the dictionary reading when the looked-up `words` senses agree on a **single** reading (unambiguous) — otherwise it trusts kuromoji's context. There is deliberately **no separate readings table** (it would only duplicate `words` readings and can't beat context). JMdict is owned by EDRDG (attribution required; UI notice deferred).

**Multi-language readiness (Japanese/Korean/Chinese and beyond).** The user-facing core — `words`, `user_words`, `lists`, `list_words`, `review_log` — is **language-agnostic by construction** and verified so (the multi-language block in `tests/integration/constraints.integration.test.ts`):
- **Language is free-form `TEXT`** (`source_lang` / `target_lang`), never an enum or FK, and it is **part of every identity key** (`words` UNIQUE includes the pair; `uq_user_words_custom` includes both langs). So a new language is just new rows — and the same surface form across pairs (愛 in JA vs ZH) stays **distinct, never collapsed**. Convention: uppercase short codes (`JA`/`EN`/`KO`/`ZH`); split scripts get their own code (`ZH-Hant` vs `ZH-Hans`). It is **unvalidated** (a `JP`/`JA` typo silently forks rows) — a DB `languages` registry + FK is the natural tightening when the 2nd language ships for real (also lets the FE `language/registry.ts` and DB agree).
- **Non-Latin storage is first-class** — inputs/translations are UTF-8 `TEXT`; Hangul (사랑) and Hanzi (爱/漢字) store, compare, and `char_length` correctly. The readings model is **two-sided + nullable** (`input_reading` / `translation_reading`): kana furigana for JA, pinyin for ZH, romanization for KO, NULL for a phonetic side — both populated only for two-logographic pairs (JA↔ZH). One phonetic annotation per side; a need for *multiple* per side (rare) would be the only reading-schema change.
- **Adding columns to existing data is cheap** — `ALTER TABLE … ADD COLUMN` is metadata-only (measured ~4 ms on the 26k-row `jmdict_kanji`), so the deferred **`frequency`** work (#7/#8: add `frequency INT` to `jmdict_kana`/`jmdict_kanji`, parse `nfXX`, `ORDER BY`) and similar are non-events. Do them as **forward-only numbered migrations** (nullable / safe-default), not by hand-ALTERing prod.
- **The Japanese-specificity is isolated to the SOURCE layer** (`jmdict_*` + `jmdict_lookup`'s `JA`/`EN` branches). Another language pairs to its OWN dictionary source (CC-CEDICT for ZH, a KR dict, …) behind the edge function's provider seam — a new `<source>_*` schema + `<source>_lookup()` + a `resolveSenseProvider` route, with `words`/`user_words` unchanged. `jmdict_glosses.lang` (DEFAULT `'eng'`) already anticipates multilingual glosses. (Locale-aware *ordering* for browse-by-alphabet — Korean dictionary order, pinyin sort — is a future collation/sort-key concern; equality lookups already work.)

## Invariants (preserve when extending `services/`)

1. **"ALL" is virtual.** A user's vocabulary *is* their `user_words` rows — there is no stored ALL list. "Every word is in ALL" is therefore **structural, not enforced in app code**. Sub-lists are optional tags; because `list_words` references a `user_words` row, a word can never be in a sub-list without being in the vocabulary. Deleting a word = deleting its `user_words` row (tags cascade). The name "ALL" stays reserved (a sub-list can't be created/renamed to it).
2. **The dictionary is server-write-only.** `words` holds only verified, system-owned senses. RLS lets clients **SELECT verified rows only** — clients cannot insert/update/delete `words` at all. Only the edge function (service role) writes verified entries. All user-authored content (created words, edits/overrides) lives in `user_words`, never in `words`.

## Module map (by responsibility)

- `config/supabaseClient.ts` — anon client singleton (throws if `VITE_SUPABASE_*` missing).
- `services/session.ts` — anonymous auth + `users` row + `getUserProfile`.
- `services/lists.ts` — sub-list CRUD (create/rename/delete/list). No ALL list; "ALL" stays a reserved name.
- `services/words/repository.ts` — **dictionary reads only**: the `words` (global cache) read API + DB-row ↔ domain `Word` (camelCase) mapping, including `inputReading` / `translationReading`. Clients never write `words`.
- `services/words/userWords.ts` — **owns `user_words` + `list_words`**: the `UserWord` type/mapping and every personal-vocabulary operation — `saveDictionaryWord`, `createCustomWord`, `editUserWord` (override), `deleteUserWord`, `addUserWordToList`/`removeUserWordFromList` — plus the display reads (`getAllUserWords`, `getUserWordsInList`, `getUserWordStates`).
- `services/entitlements.ts` — **per-user restrictions** (the limits subsystem). `getUserLimits(userId)` merges a user's `user_limits` overrides over code `DEFAULT_LIMITS` (missing row / NULL column → default); `getMonthlyUsage(userId)` reads the month-to-date char count for a "X / quota" display. Limits today: `paragraphCharLimit` + `monthlyCharQuota`. This is the **client/UX** copy — the **edge function re-enforces server-side** (the hard gate where the paid Google MT call happens). On a JMdict miss it resolves the caller's limits (JWT `sub` → `user_limits` → env → built-in default) and, BEFORE calling Google: **413**s an over-`paragraphCharLimit` paragraph, and **429**s when reserving this request's chars against `monthlyCharQuota` (`translation_usage`) would exceed it — the atomic `consume_translation_quota` reserves-before-call (no check-then-meter race). So we stay in the free tier even if the client is bypassed. Adding a money-spending feature (voice/camera) follows the same shape: a column + a field here + **server-side enforcement** at the cost point. Keep `DEFAULT_LIMITS` in sync with the edge `DEFAULT_PARAGRAPH_CHAR_LIMIT` / `DEFAULT_MONTHLY_CHAR_QUOTA`.
- `services/review.ts` — **spaced repetition**. Continuous forgetting-curve model, not a due-date schedule: `retrievability(stability, lastReviewed)` = `exp(-Δdays / stability)` (pure, tested); `getReviewQueue` ranks the vocabulary by current R ascending (the "N least-confident words"); `recordReview({ userWordId, grade })` delegates the atomic, server-clocked strength update to the `record_review` Postgres function. The grade is a **1–5 self-rated recall confidence** (1 = forgot … 5 = easy; matches the `confidence_rating 0–5` scale and the POC sketch) — there is no separate "again" (a forgotten card is grade 1; our model has no intra-session requeue). 1–2 are lapses (shrink stability), 3–5 grow it with the spacing effect. The strength UPDATE math lives in `record_review()` (init migration) and shares the `exp(-Δ/S)` curve with `retrievability` — keep in sync. Swapping to FSRS later is a new function body, not an API change.
- `services/dictionary.ts` / `services/lookup.ts` — write / read translate flows (above). `translateParagraph` overrides a token's kuromoji reading with the reading already on the looked-up `words` sense ONLY when it's unambiguous (a single distinct reading); otherwise kuromoji's context-aware reading stands. No extra query — it reuses the per-token `words` lookup it already does for meanings.
- `services/senses/` — the `SenseProvider` seam: `resolveSenseProvider(source, target)` routes a pair to a dictionary. Registry stays **empty** — the dictionary decision now lives server-side (the edge function serves JMdict), so every pair uses `mtFallbackProvider`, which delegates to the edge function and surfaces its full multi-sense `words` array.
- `services/difficulty/` — the **difficulty facade** (the DIFFICULTY axis, distinct from the future relatedness/embeddings axis — never conflate). `getDifficulty(word)` resolves a `Word` to a 1..5 level + provenance: a curated `difficultyOverride` (JLPT/HSK, normalized 1..5; NULL today) wins, else `frequency` binned per language. Mirrors the `senses/`/`language/` registry pattern — `registry.ts` routes by language (`getDifficultyJapanese` = the JMdict-nfXX bins; default reuses them since JMdict is the only frequency source), `level.ts` holds the pure mappings. PURE/read-time (no I/O), like `furiganaFor`/`retrievability`; the heavy nfXX parsing happens once upstream in the JMdict ingest.
- `services/language/` — registry (add a language = one entry in `registry.ts`), `detect.ts` (auto-detect + source resolution), `options.ts` (dropdown view-models), `tokenize.ts` (`Intl.Segmenter` word segmentation), `analyze.ts` (`analyze` → tokens + readings/lemmas; lazily-loaded kuromoji for JA, segmentation-only otherwise — the morphological-engine swap point), `furigana.ts` (`furiganaFor` → per-side reading annotations). Import via the barrel `./language`.
- `services/translation/` — the `invoke` wrapper + types (barrel `./translation`); `TranslationResult` carries `word` (primary) + `words` (all senses).
- `scripts/ingest-jmdict.ts` — one-time Node/TS loader (direct `pg`, truncate-and-reload) for the `jmdict_*` tables.
- `src/lib/concurrency.ts` — `mapLimit`; caps concurrent translate calls at `MAX_TRANSLATION_CONCURRENCY` (used by both translate flows).

## Conventions

- **snake_case DB ↔ camelCase domain**; the repository maps explicitly (no DB rows leak upward).
- **NFC-normalize user input** at every boundary (`.trim().normalize("NFC")`) — cache-key correctness, esp. Japanese. Already done in both translate flows, `customWords`, and the edge function.
- **Param style:** single primitive arg → positional; multiple args → an options object.
